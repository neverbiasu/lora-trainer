### 表 C：结论（优点、不足、可借鉴点、反模式）

| 分类 | 内容 |
| --- | --- |
| 架构优点 | 1) 模块化与接口统一：以中心抽象统一不同算法实现与集成接口，便于扩展与维护。 [[deepwiki.com]](https://deepwiki.com/KohakuBlueleaf/LyCORIS) 2) 算法覆盖面广：从 LoRA/LoCon 到 LoHa/LoKr/(IA)^3/DyLoRA 等，多目标权衡更灵活。 [[github.com]](https://github.com/KohakuBlueleaf/Lycoris), [[deepwiki.com]](https://deepwiki.com/KohakuBlueleaf/LyCORIS) 3) 多使用路径：既能集成训练器，也能独立用于 PyTorch 模块与权重工具链。 [[deepwiki.com]](https://deepwiki.com/KohakuBlueleaf/LyCORIS), [[github.com]](https://github.com/KohakuBlueleaf/Lycoris) 4) 许可证友好：Apache-2.0 便于在更广泛的生态/产品中集成。 [[github.com]](https://github.com/KohakuBlueleaf/LyCORIS/blob/main/LICENSE.md), [[github.com]](https://github.com/KohakuBlueleaf/Lycoris) |
| 架构不足 | 1) 它不是“训练器产品”：不提供像 OneTrainer/eden 那样的完整训练工作流或 UI，需要依赖外部训练框架来跑 end-to-end。 [[deepwiki.com]](https://deepwiki.com/KohakuBlueleaf/LyCORIS), [[github.com]](https://github.com/KohakuBlueleaf/Lycoris) 2) 生态集成需要版本协同：与训练器侧的加载/继续训练能力（如 create_network_from_weights）可能出现兼容依赖，需要双方配合实现。 [[github.com]](https://github.com/kohya-ss/sd-scripts/issues/577), [[kblueleaf.net]](https://kblueleaf.net/posts/lycoris-3.0.0/) |
| 可借鉴点（给你做训练器库） | 1) 将“适配器算法层”抽象成独立可替换模块，并提供统一接口/注册机制，训练器只做编排。 [[deepwiki.com]](https://deepwiki.com/KohakuBlueleaf/LyCORIS) 2) 设计多种集成通道：库 API、功能性 API、训练器集成模块、工具链 CLI，覆盖不同用户与部署场景。 [[deepwiki.com]](https://deepwiki.com/KohakuBlueleaf/LyCORIS), [[github.com]](https://github.com/KohakuBlueleaf/Lycoris) 3) 把“模型资产操作”（合并/提取/转换）做成一等公民，这对训练器产品化非常关键。 [[deepwiki.com]](https://deepwiki.com/KohakuBlueleaf/LyCORIS), [[github.com]](https://github.com/KohakuBlueleaf/Lycoris) |
| 应避免的反模式 | 1) 将训练编排与适配器实现强耦合：LyCORIS 的价值在于算法层独立可插拔；如果训练器把算法写死，后续扩展会变成复制脚本。 [[deepwiki.com]](https://deepwiki.com/KohakuBlueleaf/LyCORIS), [[github.com]](https://github.com/KohakuBlueleaf/Lycoris) 2) 版本升级不设兼容层：LyCORIS 3.x 的 API 重构说明“接口演进”会发生，训练器侧需要清晰的版本适配策略与兼容测试。 [[kblueleaf.net]](https://kblueleaf.net/posts/lycoris-3.0.0/), [[deepwiki.com]](https://deepwiki.com/KohakuBlueleaf/LyCORIS) |
