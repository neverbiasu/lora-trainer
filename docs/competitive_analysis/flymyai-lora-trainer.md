### 表 C：结论（优点、不足、可借鉴点、反模式）

| 分类 | 内容 |
| --- | --- |
| 架构优点 | 1) 配置驱动明确：训练参数集中在 `train_configs/`，方便复现与分享。 [[github.com]](https://github.com/FlyMyAI/flymyai-lora-trainer), [[github.com]](https://github.com/FlyMyAI/flymyai-lora-trainer/issues/46) 2) 数据集校验工具降低失败率：提供 `utils/validate_dataset.py` 并在实际 Issue 中被用于训练前检查。 [[github.com]](https://github.com/FlyMyAI/flymyai-lora-trainer/issues/46), [[github.com]](https://github.com/FlyMyAI/flymyai-lora-trainer) 3) 面向低显存有产品化交付：Release 描述了 <24GiB 的轻量 pipeline 与 ready-to-use 配置。 [[github.com]](https://github.com/FlyMyAI/flymyai-lora-trainer/releases), [[github.com]](https://github.com/FlyMyAI/flymyai-lora-trainer) 4) 与 ComfyUI 通过官方 workflow 对接：提供现成 workflow 文件与说明，降低推理落地成本。 [[github.com]](https://github.com/FlyMyAI/flymyai-lora-trainer), [[huggingface.co]](https://huggingface.co/flymy-ai/qwen-image-realism-lora/discussions/1) |
| 架构不足 | 1) 多脚本入口导致“脚本分裂”：不同模型/任务靠不同 `train_*.py` 承载，长期维护可能出现重复逻辑与行为不一致。 [[github.com]](https://github.com/FlyMyAI/flymyai-lora-trainer), [[github.com]](https://github.com/FlyMyAI/flymyai-lora-trainer/releases) 2) 依赖版本与生态耦合较强：requirements 固定 diffusers 到特定 Git commit，并引入 deepspeed/量化相关依赖，环境可重复性与安装稳定性可能成为痛点。 [[github.com]](https://github.com/FlyMyAI/flymyai-lora-trainer/blob/main/requirements.txt), [[github.com]](https://github.com/FlyMyAI/flymyai-lora-trainer/issues/14) 3) “兼容 ComfyUI”的边界需要更清晰：讨论区/对话中出现对 LoRA Loader 兼容性的疑问，官方倾向通过 workflow 指引来解决。 [[huggingface.co]](https://huggingface.co/flymy-ai/qwen-image-realism-lora/discussions/1), [[github.com]](https://github.com/FlyMyAI/flymyai-lora-trainer) |
| 可借鉴点（给你做训练器库） | 1) 数据集验证脚本应成为标配：训练前先验证配对文件与目录结构，能显著降低用户支持成本。 [[github.com]](https://github.com/FlyMyAI/flymyai-lora-trainer/issues/46), [[github.com]](https://github.com/FlyMyAI/flymyai-lora-trainer) 2) 配置文件应成为第一入口：用 YAML/TOML 承载可复现配置，比堆 CLI 参数更工程化。 [[github.com]](https://github.com/FlyMyAI/flymyai-lora-trainer), [[github.com]](https://github.com/FlyMyAI/flymyai-lora-trainer/issues/46) 3) 低显存路线要“成体系交付”：不仅提供参数建议，还提供完整配置与脚本入口，形成可复制方案。 [[github.com]](https://github.com/FlyMyAI/flymyai-lora-trainer/releases), [[github.com]](https://github.com/FlyMyAI/flymyai-lora-trainer) |
| 应避免的反模式 | 1) 无统一 Trainer 抽象的脚本增殖：当模型族/任务增多时，`train_xxx.py` 会越来越多，建议尽早把共性收敛到可配置的 pipeline/strategy。 [[github.com]](https://github.com/FlyMyAI/flymyai-lora-trainer), [[github.com]](https://github.com/FlyMyAI/flymyai-lora-trainer/releases) 2) 依赖被动漂移：将核心训练框架固定到特定 diffusers commit 虽能稳定当前实现，但长期可能加大维护与用户安装成本，需要清晰的版本策略与锁定机制。 [[github.com]](https://github.com/FlyMyAI/flymyai-lora-trainer/blob/main/requirements.txt), [[github.com]](https://github.com/FlyMyAI/flymyai-lora-trainer/issues/14) |
