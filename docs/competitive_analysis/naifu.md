## 表 C：结论（优点、不足、可借鉴点、反模式）

| 分类 | 内容 |
| --- | --- |
| 架构优点 | 1) **统一入口 + 配置驱动**：核心训练入口集中在 `trainer.py`，通过多份 YAML config 切换训练目标，降低 CLI 参数负担。 [[github.com]](https://github.com/Mikubill/naifu/blob/main/README.md), [[github.com]](https://github.com/Mikubill/naifu) 2) **工程分层清晰**：common/config/data/models/modules/scripts 的分区结构利于长期维护与扩展。 [[github.com]](https://github.com/Mikubill/naifu), [[github.com]](https://github.com/Mikubill/naifu/blob/main/README.md) 3) **面向研究/工程的扩展接口**：提供 LyCORIS、fairscale（以及仓库中体现的 deepspeed 支持）等路线，适合训练工程师做更复杂实验。 [[github.com]](https://github.com/Mikubill/naifu/blob/main/README.md), [[github.com]](https://github.com/Mikubill/naifu), [[deepwiki.com]](https://deepwiki.com/Nerogar/OneTrainer/7-installation-and-deployment) |
| 架构不足 | 1) **流程不够“极简一条线”**：例如 SDXL 路线要求先做 latent 编码（脚本步骤），对你想做的“最少步骤启动”是反例。 [[github.com]](https://github.com/Mikubill/naifu/blob/main/README.md), [[github.com]](https://github.com/Mikubill/naifu) 2) **主分支标注持续开发**：README 明确主分支“under development”，对稳定性与版本兼容需要额外工程治理。 [[github.com]](https://github.com/Mikubill/naifu/blob/main/README.md), [[github.com]](https://github.com/Mikubill/naifu) 3) **许可合规风险需显式处理**：issue 指出仓库 MIT 与模型许可可能不一致，实际分发/衍生需要额外合规提示与 NOTICE 策略。 [[github.com]](https://github.com/Mikubill/naifu/issues/40), [[github.com]](https://github.com/Mikubill/naifu/blob/main/README.md) |
| 可借鉴点（对你 PRD/架构） | 1) **单入口脚本 + 多 config** 的形态很契合你"CLI 参数尽量少、靠配置启动"的目标（你可以比 naifu 更进一步：让默认 config 更少字段即可跑）。 [[github.com]](https://github.com/Mikubill/naifu/blob/main/README.md), [[github.com]](https://github.com/Mikubill/naifu) 2) **分层目录结构**可作为你 Top-down 核心（TrainerCore/Adapters/Exporter）之外的工程组织参考。 [[github.com]](https://github.com/Mikubill/naifu), [[github.com]](https://github.com/Mikubill/naifu/blob/main/README.md) 3) **算法扩展路线**：naifu 用"LyCORIS 配置"来扩算法，MVP 阶段专注标准 LoRA 实现，避免后期脚本增殖。 [[github.com]](https://github.com/Mikubill/naifu/blob/main/README.md), [[huggingface.co]](https://huggingface.co/flymy-ai/qwen-image-edit-inscene-lora/blame/refs%2Fpr%2F2/README.md) |
| 应避免的反模式 | 1) 把“性能优化步骤”变成用户必须手工执行的前置流程（如强制先编码 latents）；更好的方式是提供可选缓存，并由训练器在 run 内自动管理或至少自动检测。 [[github.com]](https://github.com/Mikubill/naifu/blob/main/README.md), [[github.com]](https://github.com/Mikubill/naifu) 2) 不明确区分“代码许可 vs 模型许可”的合规边界；建议在你的 PRD/README 中明确写出 NOTICE/FAQ（issue #40 提示了真实风险）。 [[github.com]](https://github.com/Mikubill/naifu/issues/40), [[github.com]](https://github.com/Mikubill/naifu/blob/main/README.md) |
