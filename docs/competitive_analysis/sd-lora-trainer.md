### 表 C：结论（优点、不足、可借鉴点、反模式）

| 分类 | 内容 |
| --- | --- |
| 架构优点 | 1) 单训练脚本与统一损失模块同时覆盖 SD15 与 SDXL，减少“不同底模两套训练实现”的重复成本 [[github.com]](https://github.com/edenartlab/sd-lora-trainer/), [[deepwiki.com]](https://deepwiki.com/edenartlab/sd-lora-trainer/2-system-architecture)。2) LoRA 采用 PEFT，属于更标准化的生态路径，并支持 UNet 与文本编码器的 LoRA（可选） [[deepwiki.com]](https://deepwiki.com/edenartlab/sd-lora-trainer/4.2-lora-adaptation), [[deepwiki.com]](https://deepwiki.com/edenartlab/sd-lora-trainer/5-training-process)。3) 多损失与正则（token attention、DAAM、embedding reg、L1 penalty）属于偏“训练质量导向”的系统设计 [[deepwiki.com]](https://deepwiki.com/edenartlab/sd-lora-trainer), [[deepwiki.com]](https://deepwiki.com/edenartlab/sd-lora-trainer/2.2-training-pipeline)。4) 部署形态完整：CLI、ComfyUI Node、Cog/Replicate、托管服务，使训练器天然可产品化与平台化 [[deepwiki.com]](https://deepwiki.com/edenartlab/sd-lora-trainer/1.1-installation-and-setup), [[deepwiki.com]](https://deepwiki.com/edenartlab/sd-lora-trainer/8-deployment-options), [[github.com]](https://github.com/edenartlab/sd-lora-trainer/)。 |
| 架构不足 | 1) 环境依赖偏重：安装要求 NVIDIA CUDA 12.1、Python 3.11，并要求 ffmpeg（视频相关），对一部分本地环境不够友好 [[deepwiki.com]](https://deepwiki.com/edenartlab/sd-lora-trainer/1.1-installation-and-setup), [[github.com]](https://github.com/edenartlab/sd-lora-trainer/)。2) 最佳效果依赖外部 OpenAI Key 进行 caption 清洗与 token 注入，会引入外部服务依赖与可复现性问题 [[deepwiki.com]](https://deepwiki.com/edenartlab/sd-lora-trainer/1.1-installation-and-setup), [[github.com]](https://github.com/edenartlab/sd-lora-trainer/)。3) 许可证表述存在多处信号（仓库有 LICENSE.txt，pyproject 声明 GPL-3.0），若你目标是做可广泛嵌入的训练库，需要提前评估许可证兼容性 [[github.com]](https://github.com/edenartlab/sd-lora-trainer/), [[github.com]](https://github.com/edenartlab/sd-lora-trainer/blob/main/pyproject.toml)。 |
| 可借鉴点（给你做训练器库） | 1) “多入口同内核”的产品化结构：CLI、Node、Service 都只做薄封装，把训练内核固定在一套 pipeline 上 [[deepwiki.com]](https://deepwiki.com/edenartlab/sd-lora-trainer/8-deployment-options), [[deepwiki.com]](https://deepwiki.com/edenartlab/sd-lora-trainer/2-system-architecture)。2) 把“训练质量机制”模块化：将损失项与正则（token attention/DAAM/cond reg/L1）作为可配置组件，有利于后续扩展到视频/新底模 [[deepwiki.com]](https://deepwiki.com/edenartlab/sd-lora-trainer/2.2-training-pipeline), [[deepwiki.com]](https://deepwiki.com/edenartlab/sd-lora-trainer/7-hyperparameter-tuning)。3) 采用生态标准（PEFT）而非自造 LoRA 体系，可以降低维护成本与兼容性风险 [[deepwiki.com]](https://deepwiki.com/edenartlab/sd-lora-trainer/4.2-lora-adaptation), [[deepwiki.com]](https://deepwiki.com/edenartlab/sd-lora-trainer/5-training-process)。 |
| 应避免的反模式 | 1) 把 caption 质量强绑定到第三方服务密钥：更好的方式是提供可插拔 caption cleaner 接口（OpenAI 只是一个实现），并保证离线可复现路径 [[deepwiki.com]](https://deepwiki.com/edenartlab/sd-lora-trainer/1.1-installation-and-setup), [[github.com]](https://github.com/edenartlab/sd-lora-trainer/)。2) 安装门槛过高且缺少分层 requirements（例如 CPU/低配路径）：未来做库时可参考 OneTrainer 那种按硬件分 requirements 的策略 [[deepwiki.com]](https://deepwiki.com/edenartlab/sd-lora-trainer/1.1-installation-and-setup), [[kohya-ss/s...mories.com]](http://gitmemories.com/kohya-ss/sd-scripts)。 |
